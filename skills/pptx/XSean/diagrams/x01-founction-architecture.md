# 架构图通用 Spec

## 1. 目标

本规范用于沉淀一套适合 PPT 场景的通用架构图绘制方法，重点约束信息结构、版式语法、图形语法、文案粒度、连线规则和拆页条件。
本规范不包含配色要求。

## 2. 适用范围

适用于以下类型的页面：

- 平台总体架构图
- 能力地图型架构图
- 数据类平台架构图
- 管理平台架构图
- 汇报型方案总览图

不优先适用于以下类型：

- 时序图
- 详细网络拓扑图
- 微服务调用链明细图
- 算法流程图
- 实施步骤图

## 3. 核心表达原则

1. 优先用分层表达结构，不依赖复杂箭头表达逻辑。
2. 优先用分组容器表达边界，不采用自由散点式摆放。
3. 优先用模块卡片表达能力、系统或对象，不在一个模块中塞入多条信息。
4. 箭头只表达必要的方向关系，不能喧宾夺主。
5. 优先保证结构清晰，其次再补充细节信息。

## 4. 画布结构

1. 默认采用 16:9 横向页面。
2. 主体内容采用自上而下的横向分层结构。
3. 允许在页面左右两侧设置纵向标签条，用于强化总主题、约束条件或补充维度。
4. 中央区域为主信息区，优先承载核心结构。
5. 底部区域优先放输入来源、外部对象、承载对象或对接对象。
6. 如存在补充说明区、扩展区或外部系统区，宜使用独立区域承载，避免干扰主干结构。

## 5. 信息层次语法

信息层次只定义抽象语义，不预设具体业务层名。

建议按以下抽象层次组织页面：

1. 目标/上下文层
2. 能力/服务层
3. 核心对象层
4. 输入/来源层
5. 外部对象/承载层

约束说明：

1. 实际层名必须按具体业务内容重命名。
2. 不预设行业专有术语。
3. 单页建议控制在 4 到 6 层。
4. 如有外部系统对接，可放在底部或右下独立区域。

## 6. 图形语法

### 6.1 外框容器

用于表示一个域、一个层或一个相对独立的结构边界。

适用场景：

- 一整层内容的承载区
- 一个主题域
- 一个子系统边界

### 6.2 分组框

用于表示某一层内部的子域、子主题或能力组。

适用场景：

- 模块分组
- 子域归类
- 同层多组并列内容

约束：

1. 分组标题本身不带填充，也不带边框。
2. 分组标题仅作为文字标签存在。

### 6.3 圆角矩形卡片

默认基本单元，用于表示单个模块、能力点、系统项、服务项或对象项。

约束：

1. 一个卡片只承载一个核心概念。
2. 卡片之间保持统一尺寸体系。
3. 同层卡片优先统一高度。
4. X01 落地时不使用圆角矩形，统一使用直角矩形。

### 6.4 胶囊形/圆柱形模块

仅用于表示任何与数据存储相关的概念。

适用场景示例：

- 数据库
- 数据中心
- 数据仓
- 数据池
- 数据湖
- 其他存储载体

约束：

1. 不将胶囊形/圆柱形作为通用强调形状。
2. 不用于普通能力模块。
3. 仅在需要明确表达存储语义时使用。

### 6.5 虚线边框

用于表示非主干、可选、补充、外围或外部区域。

适用场景：

- 外部系统区
- 补充说明区
- 可选能力区
- 扩展能力区

### 6.6 图标

图标仅作为辅助识别元素使用。

约束：

1. 优先用于来源对象、设备对象或少量识别性对象。
2. 不建议给每个模块都配置图标。
3. 图标不能替代文字标签。

## 7. 布局规则

1. 页面整体采用横向分层，不采用自由散点布局。
2. 每层内容优先遵循“分组 > 模块”的组织方式。
3. 同层分组沿水平方向排列。
4. 同组模块沿规则网格排列。
5. 核心对象优先位于页面中部或中轴区域。
6. 承上启下的支撑条或平台条可横向贯穿一层。
7. 输入来源、采集对象、外部对象优先贴近页面底部。
8. 左右补充区不得压过中央主结构。
9. 单页主分组建议不超过 8 组。
10. 单组模块建议不超过 8 个。
11. 当内容较密时，颗粒度最大的两层标题可切换为左侧纵向标题条，以节省纵向空间。
12. 核心对象层等高密度层，模块布局应优先横向平铺，再按需要换行。
13. 层样式按内容颗粒度层级组织，而不是按页面从上到下的层序组织。
14. 每一层级都带填充色，且从颗粒度大到颗粒度小逐层加深。
15. 除颗粒度最大的层级外，其它层级默认不带边框。
16. 颗粒度最大的两层标题可使用左侧纵向标题条模式。

## 8. 文案规则

1. 层级标题使用名词短语。
2. 模块标题建议控制在 4 到 10 个字。
3. 一个模块只表达一个能力或一个对象。
4. 同层命名粒度尽量一致。
5. 说明性句子放在层间说明带、页下注释或侧边注释中。
6. 不在模块卡片内放长句说明。

命名颗粒示例仅作参考，不构成固定命名约束。

## 9. 连线规则

1. 默认不用线表达层级，层级通过空间位置表达。
2. 层间关系使用方向箭头形状表达（上箭头、下箭头、左箭头、右箭头），不使用连线。
3. 自下向上箭头表示采集、输入、支撑、汇聚。
4. 自上向下箭头表示管理、反馈、调度、控制。
5. 左右箭头表示交换、协同、流转。
6. X01 只允许层间连线，不允许模块到模块的细粒度连线。
7. 禁止大面积交叉线。
8. 如果删掉箭头后结构仍然清晰，优先删线。
9. 上下箭头放在上下两层的左右居中位置，并在两层间隙中纵向均衡居中。
10. 左右箭头放在左右两层的上下居中位置，并在两层间隙中横向均衡居中。

## 10. 标准模板

### 10.1 平台全景分层图

适用于：

- 总体方案首页
- 平台总览页
- 能力全景页

特点：

- 强调整体层次
- 适合自上而下阅读
- 用于承载最多的通用结构

### 10.2 核心对象辐射图

适用于：

- 存储核心对象为中心的结构图
- 某个中心对象连接上下游的架构图
- 中台或数据核心对象主导的架构页

特点：

- 中间一个核心对象
- 周边分布上下游模块或管理域
- 用于突出核心存储或核心承载对象

### 10.3 能力地图型矩阵图

适用于：

- 功能清单页
- 能力盘点页
- 模块规划页

特点：

- 强归类，弱流程
- 横向按域分组
- 纵向按层次或角色分区

## 11. 拆页规则

出现以下情况时，建议拆成两页或多页：

1. 主层级超过 6 层。
2. 主分组超过 8 组。
3. 单组模块超过 8 个。
4. 页面需要大量交叉连线。
5. 同时承载结构、流程、对接和明细信息。
6. 说明性文字超过 3 段。

推荐拆法：

1. 总览页讲全景结构。
2. 专题页展开单个重点域。
3. 对接页专讲外部对象和输入来源。
4. 能力页专讲模块能力和归类。

## 12. 验收标准

一张合格的架构图应满足：

1. 10 秒内可识别总主题。
2. 30 秒内可识别主要层次。
3. 1 分钟内可识别主要模块与关键关系。
4. 去掉配色后，结构依然成立。
5. 不依赖口头讲解，也能读出大致骨架。

## 13. 执行约束

1. 先定图类型，再组织内容，禁止先堆模块。
2. 先画层，再画分组，再填模块，最后补箭头。
3. 先保证对齐、分区和留白，再考虑装饰元素。
4. 每个模块只保留一个主标签。
5. 页面中最多只允许一个绝对视觉中心。
6. 连接关系不能比分层结构更抢眼。
7. 虚线区域只承担补充角色，不承担主结构角色。

## 14. 风格总结

基于样例图归纳，这套 spec 对应的表达风格如下：

1. 偏汇报型，而不是工程明细型。
2. 偏结构总览，而不是详细流程。
3. 偏模块化、卡片化、分组化表达。
4. 偏高层可读，不追求底层拓扑精度。
5. 偏一页总览，再辅以专题页展开。

## 15. Loader 输入规范（X01）

X01 对应 loader 文件：`skills/pptx/XSean/loaders/x01_function_architecture.py`

推荐输入结构：

```yaml
title: "平台功能架构总览"
subtitle: "分层 > 分组 > 模块"
content:
	side_labels:
		left: "业务视角"
		right: "治理约束"

	layers:
		- id: "context"
			name: "目标/上下文层"
			groups:
				- name: "场景目标"
					modules:
						- id: "ctx_1"
							name: "经营目标"
							type: "module"    # module | storage
		- id: "service"
			name: "能力/服务层"
			groups:
				- name: "编排能力"
					modules:
						- { id: "svc_1", name: "编排引擎", type: "module" }
						- { id: "svc_2", name: "规则中心", type: "module" }
				- name: "数据能力"
					style: "dashed"       # 可选：dashed
					modules:
						- { id: "db_1", name: "元数据库", type: "storage" }
						- { id: "db_2", name: "指标仓", type: "storage" }

	external_zones:
		- title: "外部系统区"
			style: "dashed"
			items: ["ERP", "CRM", "工单系统"]

	links:
		- { from: "context", to: "service", label: "驱动" }
		- { from: "service", to: "core", label: "沉淀" }
```

字段约束：

1. `layers` 为主结构，建议 4 到 6 层。
2. `layers[].id` 用于层间连线，建议必填。
2. 单层 `groups` 建议不超过 8 个。
3. 单组 `modules` 建议不超过 8 个。
4. `modules[].type=storage` 时会以存储语义形状渲染。
5. `links` 只用于层到层的关系，`from/to` 必须引用 `layers[].id`。
6. 层内模块默认使用主题跟随的浅填充色，不使用边框线。
7. 模块布局会根据文案长度和数量自动选择横向平铺或纵向罗列。
8. 颗粒度最大的两层支持 `title_mode: left`，将标题切换为左侧纵向标题条。
9. 所有有边框形状的边框线宽统一为 `0.5pt`，线色取自母版主题色。
10. 所有有填充的矩形内部文字默认上下、左右居中。
11. 分组标题不使用填充和边框。
